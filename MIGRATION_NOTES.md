# Migration Notes: leoroverpybullet → leorover_isaac

This document is the **file-by-file map** of the PyBullet → Isaac Lab port, so
nothing from the original stack is silently dropped. It records where every
PyBullet file went, what was carried over verbatim vs. rewritten, what is new,
the known divergences, and exactly what still needs validation on Isaac
hardware.

Read alongside `PORTING_ROADMAP.md` (the phase plan this implementation fills in).

---

## Architecture decision

The original codebase trains with **Stable-Baselines3 PPO/SAC** over 12 CPU
`SubprocVecEnv` PyBullet processes. This port targets **Isaac Lab `DirectRLEnv` +
rsl_rl PPO, GPU-vectorized to thousands of envs** — the established direction in
`PORTING_ROADMAP.md` and the right one for your goal of *fastest training on a
single 24 GB GPU*: one PhysX scene steps the whole fleet on-GPU, ~10–100× the
sim throughput of the PyBullet setup.

To keep behavioural parity with the PyBullet stack, the **single source of truth
for every hyperparameter is the original `config.py`**, carried over verbatim to
the repo root and imported by the Isaac env and agent configs. Reward weights,
residual bounds, ADR thresholds, terrain/friction ranges, and the v33.9 tuning
all stay identical and editable in one place.

## Feature-parity checklist (LeoRoverBaseEnv vs MyEnv2)

- [x] **Observation layout** — 9D pure-PPO / 11D hybrid (adds LQR cmds) / +6 camera lookahead (15D/17D). Same order: cte, heading_err, fwd_vel, lat_vel, wp_dx_body, wp_dy_body, [lqr_v, lqr_w], gx, gy, gz, [lookahead×6].
- [x] **Action** — `Box(-1,1,(2,))` residual (hybrid) or direct (v,ω) (pure PPO).
- [x] **Pure-PPO reward (v33.9, the active one)** — r_cte, r_heading, conditional r_velocity (clamped vel_scale), r_smoothness, r_alive, symmetric r_progress, hybrid-only L2 r_effort; terminal ±success/failure. Weights from `config.get_pure_ppo_reward_config()`.
- [x] **Terminations** — goal (<0.2 m), flip (g_z>0.5), out-of-bounds (goal box ±5 m), CTE-too-large (pure PPO), stagnation timeout, time limit. Same predicates.
- [x] **Waypoint skip** — advance one waypoint when the next is closer.
- [x] **Stagnation + sustained recovery** — counter with N-step sustained-recovery reset and decay-while-moving (v25/v28 logic).
- [x] **Trajectory profiling** — per-waypoint v_ref + feedforward ω_ref (curvature-based), identical to MyEnv2._compute_trajectory_profile.
- [x] **LQR baseline + bounded residual + accel limits + hardware clips** — VectorizedLQR.
- [x] **Mars gravity** — sim gravity (0,0,-3.71), as in environment2.reset().
- [x] **0.2 s policy step** — decimation=10 @ dt=1/50.
- [x] **ADR curriculum** — `ADRCurriculum` carried over; terrain difficulty rows are the ramp axis.
- [x] **episode_metrics.csv** — identical schema incl. `mean_residual_v_norm`/`mean_residual_w_norm`.
- [~] **Camera lookahead** — observation slots + binning logic present; the RayCaster/heightfield-sampling wiring is the main runtime TODO (see below).
- [~] **Counterfactual / cost / residual-RL / waypoint-offset rewards** — these are DISABLED in the current `config.py` (the active path is pure-PPO + hybrid residual, which IS ported). They are documented hooks for a follow-up if you re-enable them.

## Known divergences (and why they're acceptable)

1. **PhysX TGS vs PyBullet LCP solver.** Contact dynamics differ slightly between
   engines. PORTING_ROADMAP.md sets the realistic parity bar at **≤2 pp** success
   rate on identical seeds. `scripts/compare_hybrid_vs_lqr.py` is the test.
2. **Terrain is static after init.** PhysX colliders can't regenerate mid-run, so
   per-episode variety comes from many sub-terrain variations + respawn, not fresh
   geometry each episode (PyBullet regenerated). Slope *distribution* per difficulty
   is preserved; exact hill layout per episode differs.
3. **LQR gain K via interpolated table.** PyBullet re-solved the CARE every step;
   we precompute K(v_ref) on a 0.0025 m/s grid with the *same* scipy solver and
   interpolate — negligible difference.
4. **Wheel-radius quirk preserved.** Controller uses r=0.3 while the URDF wheels
   are ~0.06 m — kept identical to PyBullet so emergent ground speed matches.

## Now implemented (follow-up pass)

These were "remaining work" in the first pass and are now in the code:

- **Per-episode terrain variety + ADR** — `LeoRoverBaseEnv` builds an
  `ADRCurriculum` from `config.py`, accumulates per-episode success + mean CTE,
  advances/regresses the difficulty ceiling, and on **every** reset reassigns each
  env to a random terrain patch in `[0, ADR ceiling]` (random difficulty row +
  variation column). So rovers do **not** see the same hills each episode, and the
  curriculum revisits easy terrain (the original ADR design). `env.adr_stats`
  exposes the curriculum state. (Terrain geometry is still baked once at startup —
  PhysX can't regenerate mid-run — but the rover faces a fresh layout each episode.)
- **Camera lookahead** — a forward, yaw-aligned downward `RayCaster` height scanner
  (`_compute_lookahead`) fills the 6 slope features (slope/cross for near/mid/far)
  using the config camera zones; sentinel −1.0 where a zone has too few hits.
- **Friction EventTerm** — `EventCfg.wheel_friction` randomizes the wheel contact
  material over the config friction sweep each reset (contact friction =
  combine(wheel, terrain), so this reproduces PyBullet's terrain-friction sweep).
- **Experiment-launcher GUI** — `experiment_gui.py` (repo root): pick task, set
  num_envs / iterations / terrain & friction / residual bounds / reward weights /
  ADR thresholds, plus a free-form JSON box to override **any** `config.py`
  constant; launches `scripts/train.py` (or compare) via `EXPERIMENT_OVERRIDES`.

## Remaining work to first training run (ordered)

1. **Generate the USD:** `isaaclab -p scripts/convert_urdf.py` (Phase 1). Open it
   once in the GUI to confirm 4 wheel joints + sane inertias.
2. **Flat smoke test:** `isaaclab -p scripts/train.py --task Isaac-LeoRover-Flat-v0 --num_envs 64`.
   Confirm the rover drives and reward is coherent (data-path validation).
3. **API-binding shakeout (on hardware):** the Isaac-version-specific bindings to
   sanity-check on first run — `RayCasterCfg`/`GridPatternCfg` field names and the
   terrain `HfTerrainBaseCfg` sub-terrain registration in `terrain/mars_heightfield.py`.
   These are localized and version-sensitive; everything else is plain torch.
4. **Parity test:** `scripts/compare_hybrid_vs_lqr.py` vs the PyBullet baseline; aim ≤2 pp.

The full simulation, control, reward, ADR curriculum, terrain randomization,
camera lookahead, friction randomization, logging, training, comparison, and the
launcher GUI are all in place — the open items are on-hardware API/version
shakeout and the parity comparison.
